`@kb-labs/sdk` is the package plugins import from. It's a thin layer over `@kb-labs/shared-command-kit`, `@kb-labs/plugin-contracts`, and `@kb-labs/perm-presets` â€” mostly re-exports, with a handful of plugin-specific helpers on top.

The SDK groups its surface into five clusters:

1. **Handler definitions** â€” `defineCommand`, `defineRoute`, `defineAction`, `defineWebhook`, `defineWebSocket`.
2. **Runtime hooks** â€” `usePlatform`, `useLLM`, `useCache`, `useStorage`, `useConfig`, `useLogger`, `useAnalytics`, `useEmbeddings`, `useVectorStore`.
3. **Manifest helpers** â€” `defineManifest`, `defineCommandFlags`, `defineFlags`.
4. **Permission primitives** â€” `combinePermissions()`, the eight bundled presets, and the `PermissionSpec` type.
5. **Types** â€” `PluginContextV3`, `HostContext`, `ManifestV3`, `ILLM`, `ICache`, `IStorage`, and friends.

Everything you need to author a plugin lives under `@kb-labs/sdk`. You should rarely need to import from the lower-level packages directly.

## Handler definitions

One `define*` helper per host type. Each takes a definition object, validates the host at runtime, and returns a handler the sandbox runner can execute.

```ts
import {
  defineCommand,    // CLI commands (also works in workflow host)
  defineRoute,      // REST GET/POST/PUT/PATCH/DELETE routes
  defineAction,     // higher-level action endpoints
  defineWebhook,    // webhook event handlers
  defineWebSocket,  // WS channel handlers
} from '@kb-labs/sdk';
```

All five produce a handler with an `execute(ctx, input)` method. The runner imports the file specified in the manifest `handler` field, takes `module.default ?? module`, and calls `.execute(...)`. A host guard inside each helper throws if the handler runs from the wrong context â€” for example, `defineCommand` refuses anything that isn't `ctx.host === 'cli' || ctx.host === 'workflow'`.

There are corresponding type guards for runtime host checks:

```ts
import { isCLIHost, isRESTHost, isWorkflowHost, isWebhookHost, isWSHost } from '@kb-labs/sdk';
```

See [Commands](/sdk/commands), [Routes](/sdk/routes), and the other per-host pages for the full shape of each definition.

## Runtime hooks

The idiomatic way to talk to platform services from inside a handler. Every hook reads from a single global platform singleton â€” no context drilling, no dependency injection ceremony.

```ts
import {
  usePlatform,        // the whole singleton
  isPlatformConfigured,
  useLLM,             // ILLM | undefined
  isLLMAvailable,
  getLLMTier,
  useCache,           // ICache | undefined
  isCacheAvailable,
  useStorage,         // IStorage | undefined
  useVectorStore,     // IVectorStore | undefined
  isVectorStoreAvailable,
  useEmbeddings,      // IEmbeddings | undefined
  isEmbeddingsAvailable,
  useLogger,          // ILogger (always present)
  useLoggerWithContext,
  useAnalytics,       // IAnalytics | undefined
  trackAnalyticsEvent,
  useConfig,          // Promise<T | undefined>, auto-detected product scope
} from '@kb-labs/sdk';
```

All service hooks return `undefined` when the adapter isn't configured. `useLogger` is the exception â€” it always returns a logger (the platform provides a fallback). Check availability before using any service hook:

```ts
const cache = useCache();
if (cache) {
  const hit = await cache.get<Result>(key);
  if (hit) { return hit; }
}
```

See [Hooks](/sdk/hooks) for the full API of every hook.

## Manifest helpers

```ts
import {
  defineManifest,       // typed ManifestV3 builder
  defineCommandFlags,   // typed flag schemas for cli.commands[i].flags
  defineFlags,          // general-purpose typed flag schema
  generateExamples,     // build `examples: string[]` programmatically
} from '@kb-labs/sdk';
```

You can write the manifest as a plain object â€” the helpers exist to keep TypeScript honest. `defineCommandFlags` is the most commonly used one because it lets you author flags once and get both CLI parsing and type inference.

## Permission primitives

All eight presets plus the builder, re-exported from `@kb-labs/perm-presets`:

```ts
import {
  combinePermissions,   // alias for `combine()` builder
  combinePresets,       // convenience: combinePresets(a, b, c)
  minimalPreset,
  gitWorkflowPreset,
  npmPublishPreset,
  fullEnvPreset,
  kbPlatformPreset,
  llmAccessPreset,
  vectorStorePreset,
  ciEnvironmentPreset,
  presets,              // namespace with all presets
  type PermissionPreset,
  type PresetBuilder,
  type PermissionSpec,
} from '@kb-labs/sdk';
```

The builder is chainable, merges string arrays as sets, and ends with `.build()` to produce a runtime-format spec ready to drop into your manifest. See [Plugins â†’ Permissions](/plugins/permissions) for the full model.

`useLoader()` is the spinner primitive used by every first-party plugin for "Generating commit planâ€¦" style feedback. `findRepoRoot()` walks up from `ctx.cwd` until it hits a git repo â€” handy because `ctx.cwd` can be a nested subrepo in monorepo setups.

## Test utilities

```ts
import { createTestContext } from '@kb-labs/sdk';
```

A minimal `PluginContextV3` factory for unit tests. For full mock builders (stub LLM responses, assertions on platform calls, etc.), import from the `@kb-labs/sdk/testing` entrypoint instead. See [Testing](/sdk/testing).

## What's NOT in the SDK

- **`defineJob`, `defineCron`, `defineWorkflow` helpers** â€” job/cron/workflow handlers are defined the same way as other handlers but there's no dedicated helper yet. Use the underlying `CommandHandlerV3` interface directly.
- **Direct adapter construction.** If you need to instantiate an `OpenAIAdapter` with custom options, that's an adapter concern â€” import from `@kb-labs/adapters-llm-openai` directly. The SDK only re-exports interfaces, not implementations.
- **HTTP clients for the REST API.** If you're building a *product* that calls KB Labs (not a plugin that runs inside it), use the [Product SDK](/product-sdk/overview) instead.

## Versioning

The SDK entrypoint â€” `@kb-labs/sdk` â€” is backward-compatible within a major version. The current line is **2.x**; additions ship as minor releases, breaking changes wait for a major bump and get a migration note in [Migration Guide](/sdk/migration).
