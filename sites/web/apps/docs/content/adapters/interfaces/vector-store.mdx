`IVectorStore` is the semantic search backend. Combined with `IEmbeddings`, it powers RAG pipelines, code search (Mind), and anything else that needs nearest-neighbour lookup over high-dimensional vectors.

Source of truth: [core/platform/src/adapters/vector-store.ts](core/platform/src/adapters/vector-store.ts).

## Interface

```ts
interface IVectorStore {
  // Core (required)
  search(query: number[], limit: number, filter?: VectorFilter): Promise<VectorSearchResult[]>;
  upsert(vectors: VectorRecord[]): Promise<void>;
  delete(ids: string[]): Promise<void>;
  count(): Promise<number>;

  // Optional
  get?(ids: string[]): Promise<VectorRecord[]>;
  query?(filter: VectorFilter): Promise<VectorRecord[]>;
}
```

## Types

```ts
interface VectorRecord {
  id: string;
  vector: number[];
  metadata?: Record<string, unknown>;
}

interface VectorSearchResult {
  id: string;
  score: number;             // similarity, 0–1 (higher = more similar)
  metadata?: Record<string, unknown>;
}

interface VectorFilter {
  field: string;
  operator: 'eq' | 'ne' | 'gt' | 'gte' | 'lt' | 'lte' | 'in' | 'nin';
  value: unknown;
}
```

## Core methods

### `search(query, limit, filter?)`

Returns the top `limit` records most similar to the query vector, optionally filtered by metadata. Results come back sorted by score descending (highest similarity first).

```ts
const queryVec = await embeddings.embed('user authentication middleware');
const hits = await vectorStore.search(queryVec, 10, {
  field: 'language',
  operator: 'eq',
  value: 'typescript',
});
```

The `score` is normalized to `0..1` — adapters handle the mapping from their native distance metric (cosine, dot product, L2). `1.0` means identical, `0.0` means completely unrelated.

### `upsert(vectors)`

Inserts or updates records. If an `id` already exists, the existing record is replaced. Adapters should accept batches — most vector databases have bulk upsert APIs that are much faster than one-at-a-time calls.

```ts
await vectorStore.upsert([
  { id: 'chunk-1', vector: [...], metadata: { file: 'src/a.ts', line: 42 } },
  { id: 'chunk-2', vector: [...], metadata: { file: 'src/b.ts', line: 17 } },
]);
```

### `delete(ids)`

Removes records by ID. Missing IDs are not an error. Accepts an array so callers can delete in bulk.

```ts
await vectorStore.delete(['chunk-1', 'chunk-2']);
```

### `count()`

Returns the total number of records in the store. Useful for monitoring and migration checks.

```ts
const n = await vectorStore.count();
```

## Optional methods

### `get(ids)`

Bulk retrieval by ID. When absent, callers have to use `search` with a filter — which is inefficient if they already know the IDs. Adapters should implement this whenever the backend supports direct ID lookup.

### `query(filter)`

Metadata-only filtering without a query vector. Returns all records matching the filter. Used for "show me every chunk tagged with this file" kind of operations.

## `VectorFilter` in practice

The filter surface is minimal on purpose — it's a single field/operator/value triple. Adapters that support richer filter languages (Qdrant's nested filters, Pinecone's filter objects) should accept the simple triple and translate it. Callers that need richer filtering should drop down to the adapter-specific API.

## Contract rules

- **Scores are normalized to `0..1`.** Higher is more similar. Adapters handle the mapping.
- **Upserts are idempotent.** Calling `upsert` with the same record twice is fine.
- **Deletes are idempotent.** Missing IDs don't throw.
- **Metadata is opaque.** The store round-trips it without interpretation (except for filtering).
- **Vector dimensions must match the index.** Adapters typically enforce this — passing a mismatched vector throws a clear error.

## Writing an `IVectorStore` adapter

```ts
import type { AdapterManifest, AdapterFactory, IVectorStore, VectorRecord, VectorSearchResult, VectorFilter } from '@kb-labs/core-platform';

export const manifest: AdapterManifest = {
  manifestVersion: '1.0.0',
  id: 'my-vector-store',
  name: 'My Vector Store',
  version: '0.1.0',
  type: 'core',
  implements: 'IVectorStore',
  capabilities: { search: true, batch: true },
};

class MyVectorStore implements IVectorStore {
  private records = new Map<string, VectorRecord>();

  async search(query: number[], limit: number, _filter?: VectorFilter): Promise<VectorSearchResult[]> {
    const scored = Array.from(this.records.values()).map(r => ({
      id: r.id,
      score: cosineSimilarity(query, r.vector),
      metadata: r.metadata,
    }));
    return scored.sort((a, b) => b.score - a.score).slice(0, limit);
  }

  async upsert(vectors: VectorRecord[]): Promise<void> {
    for (const v of vectors) this.records.set(v.id, v);
  }

  async delete(ids: string[]): Promise<void> {
    for (const id of ids) this.records.delete(id);
  }

  async count(): Promise<number> {
    return this.records.size;
  }

  async get(ids: string[]): Promise<VectorRecord[]> {
    return ids.map(id => this.records.get(id)).filter((v): v is VectorRecord => !!v);
  }
}

function cosineSimilarity(a: number[], b: number[]): number {
  let dot = 0, na = 0, nb = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    na += a[i] * a[i];
    nb += b[i] * b[i];
  }
  return dot / (Math.sqrt(na) * Math.sqrt(nb));
}

export const createAdapter: AdapterFactory<unknown, {}, IVectorStore> = () => new MyVectorStore();
```

Real adapters use proper ANN indexes (HNSW, IVF) rather than scanning every record.
